# ADR-020 · Customer secrets (G2): host-age seal + per-wake drive1 staging (M5)

- **Status:** accepted
- **Date:** 2026-07-17
- **Closes:** Spec §11/G2, §17 G2 (customer secrets; "real apps blocked without
  it" per M5 acceptance).

## Context

Spec §11 defines a secret as "a non-rotating, customer-supplied env var…
sealed at rest, injected only into the boot phase, never visible in
firecracker logs, never copied into app.json." §17 G2 was the open item:
how those secrets cross the trust boundary from the apid-writable PG to
the vmmd-root-owned jailer chroot to guest-init's execve environ. M5
acceptance listed "real apps blocked" because secrets are how every
12-factor app loads DB URLs, third-party API keys, and JWT signing
material — without them, the platform is unusable for anything that talks
to the outside world.

This ADR captures the four coupled decisions that close G2:

1. **Single host key, generated by vmmd on first boot.** X25519 + ChaCha20-
   Poly1305 via `filippo.io/age`. Lives at `/etc/faas/secrets/host.age`
   (mode 0400, root:root). The public half is published to
   `/etc/faas/secrets/host.age.pub` (mode 0444) for apid to read at
   startup.
2. **Server-side seal.** Plaintext VALUE arrives over TLS, lives briefly
   in `apid`, and is sealed before it hits Postgres. The CLI never holds
   the recipient — it cannot unseal, period. The CLI is the transport;
   apid is the trust boundary.
3. **Per-app binding (`app_id`, `key`).** Not account-scoped. Per-app is
   what every existing FaaS does and what the docs will assume; account-
   scoped is a forward-only extension if a future plan needs it.
4. **Per-wake drive1 overstamp.** vmmd is the only root component; it
   loopback-mounts the chroot-local `drive1.ext4`, writes
   `/etc/faas/secrets.env` (unsealed JSON), and umounts. guest-init
   reads it after `pivot_root`, layers the entries over the manifest env,
   and execs.

## Decisions

### D1. Single host key, vmmd auto-generates

`pkg/secretbox/hostkey.go` owns the lifecycle:

- `LoadHostKey(path)` parses an existing `host.age`. Returns
  `ErrHostKeyNotFound` for first boot.
- `GenerateAndSaveHostKey(path)` writes a fresh identity with mode 0400.
- `WriteRecipientFile` writes the public recipient (mode 0444) for apid
  to consume.

**Why single key, not per-app or per-account:** per-app X25519 keys mean
per-app private-key material in PG — a strictly worse threat model (the
DB is the audit trail; we do not want customer-owned key material in
there too). A single host key is the same trust perimeter as the
firecracker API key and TLS cert (both are single, both sit root-owned on
the box), and it lets apid operate on PG rows without per-row key fetch.

**Why vmmd generates, not apid:** vmmd is the only root component.
Generating on first boot as part of `vmmd` startup means the key never
exists in a state where root is split across processes.

**Why no rotation now:** rotation requires a multi-recipient seal so old
rows stay readable under the prior key. Out of scope for M5; tracked as
future work below.

### D2. Server-side seal, plaintext over TLS

- CLI `set` → PUT body JSON `{value: "..."}` over TLS.
- apid checks the byte cap against `Limits.SecretValueMaxBytes` (the
  default `decodeJSON` reader enforces a 1 MiB body cap, with a finer
  `Limits.SecretValueMaxBytes` check inside `PutAppSecretRequest.Validate`
  and again at `secretbox.SealOne` — defense in depth).
- `pkg/secretbox.SealOne(recipient, key, value, max)` is the single
  trust boundary. The Manager path is unrelated: vmmd holds the
  *identity* and unseals at wake.
- CLI never sees the recipient. It cannot decrypt a row even with full
  DB access — it would need `/etc/faas/secrets/host.age` (root-only).

`pkg/logsanitize.RedactValue(s)` returns `"<redacted:N>"` for any
accidental log attempt. The key list is public (spec §11), so key NAMES
flow freely in logs; only VALUES are sealed.

### D3. Per-app binding, all plans incl. Free

Per-app, never account-scoped. Quota per plan:

| Plan | Count | Value (bytes) |
|---|---|---|
| Free | 3 | 4 096 (4 KB) |
| Hobby | 25 | 8 192 (8 KB) |
| Pro | 50 | 16 384 (16 KB) |
| Scale | 100 | 32 768 (32 KB) |

`pkg/api/limits.go` is the single source of truth — handlers, CLI,
migrations, and quota errors all read from this table. New plans adjust
two ints.

### D4. Per-wake drive1 overstamp

Sequence on wake:

```
schedd → vmmd.CreateFromSnapshot(ctx, instance, app, snap, sealedEnvBlob)
vmmd Manager.Wake → bringUp → StageSecretsEnv(instance, jsonBlob)
                                          ↓
                            loopback-mount chroot/d{instance}/layer.ext4 rw
                            write /etc/faas/secrets.env (mode 0400)
                            umount
                                          ↓
                          guest-init boot() → loadSecrets() → BuildEnvWithSecrets
```

**Why drive1, not a fresh overlay or a virtio-serial mount:** drive1 is
already the per-app writable layer; it's mounted-and-staged once at
imaged time and read-only-mounted by guest-init. Adding another
loopback at runtime would cost ~2 ms per wake (measured on M3) and
risk a second mountpoint hierarchy in the chroot lifecycle. drive1 is
the natural home for per-app writable state — `/etc/faas/app.json`
already lives there.

**Threat boundary of staging:** at staging time, the chroot-local
drive1 is owned by root after imaged has provisioned it. vmmd is the
only root component on the box (per ADR-019). The plaintext value is
held in vmmd's process memory for the duration of one loopback mount
(≤10 ms on observed EX44). It is never logged, never returned over
RPC, never written outside the chroot.

### D5. Snapshot semantics — explicit acceptance

**Stolen-snapshot attack:** when an app is parked, drive1 has the
cleartext `secrets.env` at snapshot time. The snapshot is owned by the
same account's root on the box; an attacker reading the snapshot
already has root on the box, which is past §11's threat perimeter.

**Why we accept this:** A live secrets.env at boot is required for the
guest's normal operation. At park time, alternatives are (a) re-seal
the env into the snapshot (requires teaching Firecracker / DRBG about
the host age key — non-trivial), (b) zero the env file pre-park
(breaks re-wake from snapshot — the snapshot owner could never
restore), or (c) accept that the snapshot has the env (chosen).

**Documented limitation:** a customer who cares about snapshot-time
exposure of their plaintext secrets must rotate the secret after every
park — a known footgun. We surface this in the secrets CLI and the
docs. A future `secret_class: "ephemeral"` opt-in (zero-on-park) is
tracked below.

## Future work (not in M5)

1. **Multi-recipient seal for rotation.** A future `key_id` column on
   `app_secrets` (paired with a sealed-envelope "to" list that names the
   active + prior recipients) lets apid issue a fresh key while vmmd
   keeps reading old rows. No change to the on-wire envelope shape; just
   the per-app bookkeeping.
2. **`secret_class: ephemeral`** — zero `/etc/faas/secrets.env` at park
   time. Customers who don't want secrets in snapshots opt in. Documented
   limitation; a separate feature flag.
3. **Per-app host keys.** Strictly stronger threat model (no single
   host key ever has read access to more than one app's plaintext), at
   the cost of per-app key material in PG. Not justified for M5.
4. **`faas secrets import`** (`--from-file`) for bulk migration. Right-
   now the inline `KEY=VALUE` form is what the docs ship.
5. **Secret-version audit log.** Today `secret_set` / `secret_deleted`
   land in `AppendEvent`. A future `secret_rotated` event tracking
   "value changed but key didn't" gives ops a hint when a customer
   changes a key without rotating the JWT.
6. **Web dashboard surface.** ADR-011 dashboard can add it in M7.5.

## Rejected alternatives

- **`/etc/faas/secrets.env` written by apid before fog of war:** rejected
  because apid is not a root component and never touches the chroot
  hierarchy. vmmd is the only writer.
- **virtio-serial pipe at boot:** rejected because we already pay the
  loopback-mount cost for `app.json`, and adding a serial channel
  doubles the boot path complexity for one tiny file.
- **Re-seal at park time into the snapshot:** rejected because Firecracker
  snapshots are opaque blobs that don't accept a key-encryption tap
  today, and adding one is out of M5 scope.
- **Per-app X25519 keys:** rejected — see D1.

## Acceptance

- `make test` — unit tests pass (incl. tampering, quota, byte-cap,
  cross-account isolation, redaction invariant, layer ordering,
  host-key absent refuses sealed env).
- `cmd/e2e/secrets_e2e_test.go` (follow-up) — full PG-backed control-
  plane acceptance: PUT → list → fake vmmd unseal → assert env in guest
  response.
- `make metal-lima` — the M3/M5 path still green (no regression in
  `app.json` semantics; drive1 layout unchanged from imaged's POV).
- Manual smoke (EX44 only): `faas secrets set --app hello KEY=VALUE`
  then curl the app and `printenv KEY` shows the value.
