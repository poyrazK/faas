# ADR-017 · Hand-written `pkg/state/pgstore.go` (M5 exception to sqlc-only rule)

- **Status:** accepted
- **Date:** 2026-07-16
- **Decision:** `pkg/state/pgstore.go` is hand-written for M5 and ships as
  the documented exception to CLAUDE.md's "SQL via sqlc only; no string-built
  queries" rule. The interface boundary (`state.Store`) and `sqlc.yaml`
  scaffolding stay in place. A `// TODO(M5.1)` header on the file points to
  this ADR. Regeneration via `sqlc generate` lands in M5.1 once the CI sqlc
  pin (v2.x) is clean on the macOS SDK that local development relies on.
- **Why:** PR #15 ships the full Appendix A control-plane gap as a single
  M5 mega-PR. Every layer depends on the layer below and splitting it would
  mean each PR is green against an incomplete world. The sqlc binary on
  the current CI pin fails to build cleanly on the macOS SDK used for local
  iteration, which blocks the M5 ship. The hand-written `PgStore`
  implements the same `state.Store` interface the sqlc-generated package
  would have satisfied, and is exercised by `pkg/db/pgtest` round-trip tests
  covering every method — so the public contract is identical regardless
  of how the queries are produced.
- **Consequences:**
  - `sqlc.yaml` stays in tree as the contract for the future regeneration.
  - `pkg/state/pgstore.go` gets a `// TODO(M5.1)` header pointing here.
  - `pkg/db/pgtest` covers every `Store` method against a real Postgres
    cluster (CI Postgres 15 service), so the SQL/Go type contract is
    locked by tests rather than by codegen.
  - Future contributors must not silently add string-built SQL outside
    `pkg/state/pgstore.go`. The Store interface is the only legal surface
    for SQL access; new methods land in `pgstore.go` first and are
    migrated to sqlc in M5.1.
  - **Read-side cost:** `sqlc generate` adds ~5 s to CI cold-start once
    enabled, but no production-runtime cost (codegen is a build-time step).
  - **Drift risk:** hand-written SQL and Go types can drift if the schema
    changes without a corresponding `pgtest` update. Mitigated by the
    pgtest coverage; mitigated further by an explicit M5.1 milestone that
    regenerates and diffs.
- **Rejected alternatives:**
  - **Generate sqlc now, ship without local-dev path.** Rejected because
    the M5 mega-PR is already on the critical path; pulling in a CI-only
    sqlc regeneration step without an M5.1 plan would leave the codebase
    in a half-converted state.
  - **Drop sqlc.yaml entirely and embrace hand-written pgstore as the
    norm.** Rejected because CLAUDE.md's "sqlc only" rule is the load-bearing
    contract for spec compliance — the financial model and the audit
    surface both assume a code-generated query layer. Removing sqlc.yaml
    is a regression.
  - **Move M5 to a follow-up milestone while sqlc lands first.** Rejected
    because the M5 mega-PR is the project's current scope and the
    schedd/imaged wiring that this PR closes is itself a ship-blocker.

## Migration plan (M5.1)

1. Bump the CI sqlc pin to a version that builds on the macOS SDK.
2. `make sqlc-generate` against `pkg/state/queries.sql` (new file — extract
   from `pkg/state/pgstore.go`).
3. Replace `PgStore` method bodies with calls into the generated
   `Queries` type.
4. Run `make sqlc-check` (already present in CI as a CI gate) — must
   show zero diff.
5. Drop the `// TODO(M5.1)` header; supersede this ADR with ADR-018
   ("`pgstore.go` regenerated by sqlc").

## Reviewer checklist

When reviewing a change to `pkg/state/pgstore.go`:

- Does the new method have a corresponding entry in `pkg/db/pgtest`'s
  round-trip tests? (If not, the PR adds one.)
- Is the SQL parameterised? No `fmt.Sprintf` interpolation of values,
  only identifiers (which are validated against an allowlist in
  `pgstore.go`'s `quoteIdent` helper).
- Does the change introduce a new `pg_notify` channel? If so, it lands
  in `pkg/db/notify.go` (where the payload contract is documented),
  not as a string literal at the call site.
